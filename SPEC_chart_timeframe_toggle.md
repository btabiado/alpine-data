# SPEC — Chart timeframe toggle (Daily / Weekly / Monthly + YTD) on ETF charts

**Status:** SPEC ONLY — not built. Captured 2026-06-02 (from the KWG chat by mistake; build later in the ETF-dashboard chat).
**Project:** `~/btc-eth-etf-dashboard` — **PRODUCTION** (V1 ships at `/`, V2 at `/v2/` on every push to `main` via `pages.yml`).

## Goal / user story
When viewing a crypto (BTC/ETH) — and ideally a specific fund/ticker — let the user switch the chart **timeframe**: granularity **Daily / Weekly / Monthly**, plus a **YTD** (and 1Y / All) date range. Today the ETF flow charts effectively show one default view.

## ⭐ TL;DR — most of this already exists (it's a re-surface, not a new build)
Recon of V1 `app.py` (2026-06-02):
- **Data is READY.** The ETF flow builder already emits `daily`, `weekly` (W-MON resample), `monthly` (MS), `yearly` (YS) series + a `ytd` scalar, plus `by_fund` / `by_fund_daily` (~lines 122–227). Same structure in V2 `v2/app.py` (~133–144).
- **State is READY.** `const state = { tab:'etf', asset:'btc', period:'daily', range:'all', fundwin:'30', macroRange:'1Y', … }` (~3565).
- **Renderers ALREADY switch by period + range.** `flowChart` and `cumChart` do `const series = applyRange(d[state.period])` (~3881, ~3890); `applyRange()` maps `state.range`→day-count, `'all'`→null (~3647); a `resample()` / `ra(rows, aggBy)` helper exists (~3707).
- **Handlers ALREADY exist.** Click handlers bind `[data-period]` → `state.period` → `renderAll()` and `[data-range]` → `state.range` → `renderAll()` (~11223–11226), with `setActive(group, val)` toggling the active-button class.
- **What's MISSING:** just the **UI buttons.** A global Period+Timeframe header bar was deliberately **removed as clutter** (comment at ~1593–1597), leaving state + renderers + handlers intact but nothing to drive them.

➡️ **So the core work = re-add a small per-chart button row (`data-period` / `data-range`) + one `'ytd'` case in `applyRange()`.** Verify first that no `[data-period]`/`[data-range]` buttons remain in the markup (grep `data-period=` / `data-range=`).

## The one real gap: YTD
`applyRange()` (~3647) maps `state.range` to a **fixed day-count** (`map[state.range]`), `'all'`→null. **YTD is not a fixed window** — it's "since Jan 1 of the latest data year." Add a `'ytd'` branch to `applyRange()`: keep rows where `date >= Jan-1 of max(date).year` (mirrors the server-side `ytd` calc at `app.py` ~198: `df[df["date"].dt.year == max-year]`). Granularity (Daily/Weekly/Monthly) then composes automatically.

## Scope decisions (confirm before building)
1. **V1, V2, or both?** V1 is production-protected; V2 is newer. **Source-of-truth:** V1 = the HTML/JS template string inside `app.py`; V2 = `HTML_TEMPLATE` inside `v2/app.py`. **NEVER hand-edit the built `dashboard.html`** (regenerated by the app). Recommend prototyping in **V2** (or locally), then porting to V1.
2. **Which charts?** ETF Flows (`flowChart` ~2058, `cumChart` ~2062) are fully wired for period+range today → cheapest, do these first. The **price chart** (`priceChart` ~2174 / built ~4023, uses the `a.price` daily/hourly series ~3927) is NOT part of the daily/weekly/monthly flow aggregation — to give it a period toggle, route its series through the existing `resample()` by `state.period`. Flag as separate/optional.
3. **"crypto or stock":** asset-level BTC/ETH already exists via `state.asset` (`etfAsset()` / `accentFor()`). Per-fund ("stock" = individual ETF tickers like IBIT/FBTC) has `by_fund` / `by_fund_daily` (daily only) — weekly/monthly per-fund would need server-side aggregation; decide if in scope.
4. **Granularity vs range** are two axes the phrase "weekly/monthly/YTD" mixes. Recommend ONE compact row per chart: `[D] [W] [M]` (`data-period`) + `[YTD] [1Y] [All]` (`data-range`), reusing the existing pattern + `setActive`.

## Implementation sketch (frontend-mostly)
1. Add a control row above `flowChart`/`cumChart` mirroring the existing `data-macrorange` (~2273–2277) / `data-fundwin` (~2087–2090) button rows: `<button class="btn" data-period="daily">D</button>` …`"weekly"`/`"monthly"`, and `data-range="ytd"`/`"1y"`/`"all"`.
2. Add the `'ytd'` branch to `applyRange()`.
3. Confirm `setActive('period', …)` / `setActive('range', …)` highlight the new buttons (generic-by-group, should just work).
4. (Optional) Wire `priceChart` through `resample(series, state.period, …)`.
5. Decide per-chart vs per-ETF-tab-shared control (the removed bar was *global* = clutter; per-chart or per-tab-local is likely better UX).

## Acceptance criteria
- Weekly/Monthly re-buckets flow + cumulative; YTD limits to since-Jan-1; All = full history; active button highlights.
- BTC + ETH each respect the selected timeframe; defaults unchanged (`daily` / `all`) so nothing regresses.
- Weekly sum reconciles to the daily series; YTD reconciles to the server `ytd` scalar.
- 0 console errors; both V1 and V2 (if both in scope) verified after a hard-refresh.
